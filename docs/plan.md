# Atom Plan

## Goal

Build a Rust-first framework for shipping iOS and Android apps with:
- App code written in Rust.
- A headless shared runtime for state, effects, navigation, and module access.
- Continuous native generation (CNG) that derives native host code from config and module metadata.
- Native modules authored in Rust, with Swift/Kotlin glue generated by the framework.

This is intentionally "headless first." The foundation should not depend on a renderer. A UI layer can be added on top once the runtime, CNG, and native-module system are stable.

## Constraints

- Bazel is the source of truth for builds, tests, and code generation.
- Use `bzlmod` only. Do not introduce `WORKSPACE`.
- Pin `.bazelversion` to `8.4.2`.
- Manage developer tooling through `mise.toml`.
- Mirror the Forge Rust setup:
  - `rules_rust` via `MODULE.bazel`
  - `rust.toolchain(edition = "2024", versions = ["1.92.0"])`
  - One `third-party/rust/Cargo.toml` and `Cargo.lock`
  - `crate_universe` as the Bazel entrypoint for Rust dependencies

## Product Shape

The target developer experience is:
1. Write an app crate in Rust.
2. Declare app config in `Atom.toml`.
3. Add Rust native-module crates to the app.
4. Run `atom prebuild` to generate iOS and Android host code.
5. Build and run with Bazel-driven commands.

The app author should not need to write Swift, Kotlin, Gradle, or Xcode project files directly. Native code will still exist, but it should be generated, minimal, and framework-owned.

## Non-Goals For V1

- A full cross-platform widget renderer in the first slice.
- Manual editing inside generated native directories.
- OTA updates, live reload, or Expo-style dev server features before the core runtime works.
- Desktop or web support before mobile foundations are proven.

## Architecture

### 1. App crate

Each app is a normal Rust crate. It defines:
- The app entrypoint.
- Routes or screen descriptors.
- Shared state and async effects.
- The set of modules it uses.
- Environment-specific config hooks.

### 2. Headless runtime

The runtime owns:
- App lifecycle.
- Event dispatch.
- State transitions.
- Async task execution.
- Navigation state.
- Calls into native modules.

This should be platform-agnostic Rust with no iOS/Android logic mixed into it.

### 3. Module SDK

A native module is a Rust crate that exposes:
- A stable module identifier.
- A typed Rust API used by the app.
- A manifest that declares permissions, platform requirements, config fragments, and generated bridge needs.
- Optional platform-specific implementation crates when a module truly needs them.

The framework should aggregate module manifests during CNG and use them to shape the generated native host.

### 4. Bridge layer

The bridge is the boundary between Rust and generated Swift/Kotlin shims.

Planned direction:
- Use a framework-owned ABI instead of a general-purpose bridge like JS or React Native.
- Keep Swift/Kotlin very thin: bootstrap, lifecycle forwarding, and marshaling.
- Generate the registration tables from Rust module metadata.

This favors a custom C ABI over `uniffi` for the core framework boundary because the framework controls both ends and needs deterministic startup, lifecycle, and registry behavior. `uniffi` can still be evaluated later for isolated module APIs if it reduces boilerplate.

### 5. CNG engine

CNG should take:
- `Atom.toml`
- App crate metadata
- Module manifests
- Build profile / environment data

And deterministically generate:
- `generated/ios/...`
- `generated/android/...`
- Platform manifests and config files
- Swift/Kotlin registration glue
- Bazel targets or Bazel-readable source trees for the generated hosts

The generated output must be idempotent. Running CNG twice with the same inputs should produce the same tree.

## Build And Tooling Plan

### Bazel

Planned root files:
- `.bazelversion`
- `MODULE.bazel`
- `MODULE.bazel.lock`
- `BUILD.bazel` only where needed for real targets

Planned module setup, based on Forge:
- `bazel_skylib`
- `platforms`
- `rules_rust`
- Rust toolchains registered through the `rules_rust` extension
- `crate_universe` fed by `third-party/rust/Cargo.toml` and `Cargo.lock`

Additional Bazel dependencies will likely be needed during implementation:
- `rules_apple`
- `rules_swift`
- `apple_support`
- Android rules compatible with Bazel `8.4.2`

I do not want to pin those yet in the plan because the compatibility matrix should be validated against the exact local Xcode and Android SDK setup when we implement.

### Rust dependency model

Follow the Forge layout exactly in spirit:

```text
third-party/rust/
  BUILD.bazel
  Cargo.toml
  Cargo.lock
```

That virtual Cargo package exists only to define dependencies imported by `crate_universe`. Application and framework crates then depend on Bazel-generated `@crates//...` targets rather than running Cargo directly in normal development.

### mise

Planned `mise.toml` responsibilities:
- Pin `bazel = "8.4.2"`
- Pin `bazelisk`
- Pin `rust = "1.92.0"`
- Optionally pin `cargo-binstall` or other support tools later, but not in the initial slice

Local validation in this environment shows `mise` has registry entries for:
- `bazel`
- `bazelisk`
- `rust`

That means the toolchain plan is viable without inventing custom plugin wiring.

Notes:
- `atom-manifest` owns `Atom.toml` parsing and validation.
- `atom-cng` resolves the app + module graph into generated native output.
- `atom-ffi` owns the stable host ABI.
- `atom-cli` is a thin frontend over CNG and Bazel commands.

## CNG Model

### Inputs

`Atom.toml` should describe:
- App identity and bundle identifiers.
- iOS and Android deployment targets.
- Assets and icons.
- Feature flags.
- Build profiles.
- Declared modules.

Each module should contribute:
- Required permissions.
- `Info.plist` fragments.
- Android manifest fragments.
- Entitlements.
- Native initialization hooks.
- Additional generated source requirements.

### Pipeline

1. Parse and validate `Atom.toml`.
2. Resolve the Rust app crate and its module list.
3. Load module manifests.
4. Merge platform requirements into one generation graph.
5. Emit deterministic native host trees.
6. Build those trees with Bazel.

### Ownership rule

Anything under `generated/ios` and `generated/android` is framework-owned. If users need customization, they should express it through config plugins or module metadata, not ad hoc file edits.

## Native Module Model

Each module should have three layers:
1. A Rust-facing API used by the app.
2. A manifest that describes platform requirements.
3. A platform bridge implementation used by generated host code.

The key requirement is that module authors stay mostly in Rust. Swift/Kotlin should be generated or narrowly isolated to framework-maintained shims.

Good first-party modules for the first usable milestone:
- `device_info`
- `secure_storage`
- `file_system`
- `http`

Avoid camera, push notifications, and background execution in the first pass. They multiply native surface area too early.

## UI Strategy

Because you asked for a headless framework, the foundation should stop short of a renderer at first.

The clean sequence is:
1. Build the runtime, CNG, and native-module layer.
2. Prove that a Rust app can boot inside generated iOS and Android hosts.
3. Add a renderer as a separate concern.

The likely renderer direction is a Rust-authored declarative view tree that diff-applies native widget commands through the same bridge. That should be treated as Phase 2, not bundled into the first bootstrap.

## Phased Implementation Plan

### Phase 0: Toolchain bootstrap

Deliverables:
- `mise.toml`
- `.bazelversion` pinned to `8.4.2`
- `MODULE.bazel` using Forge-style `rules_rust` setup
- `third-party/rust/{Cargo.toml,Cargo.lock,BUILD.bazel}`
- One trivial Rust target built by Bazel on the host machine

Exit criteria:
- `mise install` works
- `bazel test //...` works for at least one Rust target

### Phase 1: Manifest and dry-run CNG

Deliverables:
- `Atom.toml` schema
- `atom-manifest`
- `atom-cng`
- `atom-cli`
- `atom prebuild --dry-run`

Exit criteria:
- Dry-run output shows the full iOS/Android generation plan from a sample app

### Phase 2: Bootable native hosts

Deliverables:
- Generated iOS host with Swift bootstrap
- Generated Android host with Kotlin bootstrap
- Rust staticlib or cdylib integration
- One sample module: `device_info`

Exit criteria:
- Example app launches on iOS simulator and Android emulator
- Rust app lifecycle callbacks execute on both platforms

### Phase 3: Core runtime

Deliverables:
- State container
- Event/effect model
- Navigation state
- Async runtime integration
- Structured logging

Exit criteria:
- Example app performs real state changes and module calls on both platforms

### Phase 4: Developer workflow

Deliverables:
- `atom run ios`
- `atom run android`
- `atom test`
- Config plugin story for native customization

Exit criteria:
- A new app can be created, prebuilt, and run with one documented workflow

### Phase 5: Optional renderer

Deliverables:
- View tree abstraction
- Native widget diff/apply layer
- Layout and styling model

Exit criteria:
- A simple UI app can be authored entirely in Rust

## Open Questions

- Do we want generated Xcode projects directly, or should Xcode be a derived artifact from Bazel later via `rules_xcodeproj`?
- Should the core Rust output be a `staticlib`, `cdylib`, or both?
- Do we support Android on Linux from the start, or only macOS first while iOS is being brought up?
- How much of app configuration lives in `Atom.toml` versus Rust macros?
- What is the minimum ABI surface needed to keep the bridge stable as modules evolve?

## Recommended First Implementation Slice

Once planning is approved, the first build work should be deliberately narrow:
1. Toolchain files only: `mise.toml`, `.bazelversion`, `MODULE.bazel`, and `third-party/rust`.
2. A tiny `atom-manifest` crate that parses `Atom.toml`.
3. A tiny `atom-cli` crate with `prebuild --dry-run`.

That gives us a real Bazel + Rust base, validates the developer setup, and lets us prove the CNG graph before we touch iOS or Android host generation.
